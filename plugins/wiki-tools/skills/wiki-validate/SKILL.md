# Wiki Validate

위키 전체의 콘텐츠 품질, 신뢰성, 논리적 일관성을 점검하고 체크리스트 보고서를 생성한다.
**보고서 먼저 출력 → 사용자 승인 → 수정 진행** 순서를 반드시 지킨다.

## Phase 1 — 스캔 준비

1. 현재 작업 디렉토리에서 `index.md`를 읽어 전체 페이지 목록을 파악한다.
2. `log.md`의 최근 10개 항목으로 최근 변경 맥락을 파악한다.
3. 점검 범위를 사용자에게 확인한다 (기본값: 전체 위키):
   - 전체 위키
   - 특정 카테고리만 (sources / entities / concepts / syntheses)
   - 특정 태그를 가진 페이지만

## Phase 2 — 구조적 체크 (자동, bash 기반)

다음을 `find`와 `grep`으로 자동 탐지한다.

**A. Orphan 페이지** — 어떤 페이지에서도 wikilink로 참조되지 않는 페이지
```bash
# 모든 wikilink 수집
grep -rh '\[\[' wiki/ --include="*.md" | grep -oP '(?<=\[\[)[^\]|]+' | sort -u > /tmp/linked_pages.txt
# wiki/ 아래 파일명 목록과 비교해 미참조 파일 탐지
find wiki/ -name "*.md" | xargs -I{} basename {} .md | sort > /tmp/all_pages.txt
comm -23 /tmp/all_pages.txt /tmp/linked_pages.txt
```

**B. `## 출처` 섹션 누락** — concept/entity 페이지에서 출처 섹션이 없는 경우
```bash
find wiki/concepts wiki/entities -name "*.md" | xargs grep -L "^## 출처"
```

**C. Frontmatter 결함** — type/created/updated/tags 중 하나라도 없는 페이지
```bash
find wiki/ -name "*.md" | while read f; do
  if ! grep -q "^type:" "$f" || ! grep -q "^created:" "$f" || \
     ! grep -q "^updated:" "$f" || ! grep -q "^tags:" "$f"; then
    echo "$f"
  fi
done
```

**D. 시점 의존 표현** — 날짜·버전 참조, "최신", "현재" 등
```bash
grep -rn "2024\|2023\|2022\|최신\|현재\|곧\|출시 예정\|다음 버전\|beta\|v[0-9]\+\.[0-9]" \
  wiki/ --include="*.md" -l
```

**E. 기존 모순·불확실 표기 목록** — 미해결 상태 추적
```bash
grep -rn "⚠️ 모순\|❓ 불확실\|🕰️ 시점" wiki/ --include="*.md"
```

## Phase 3 — 콘텐츠 품질 체크 (LLM 판단)

구조 체크 후, `index.md`의 카테고리 순서대로 페이지를 샘플링해 읽는다.
한 번에 전체를 읽지 않는다 — 카테고리별로 묶어 순서대로 처리한다.

각 페이지에 대해 판단:

1. **근거 없는 단언** — `sources` frontmatter나 `## 출처`에 뒷받침 없이 사실을 단정하는 문장
2. **논리적 비약** — 전제-결론 사이에 설명되지 않은 도약
3. **내부 모순** — 같은 페이지 내에서 서로 다른 주장
4. **교차 페이지 모순** — 같은 개념을 다루는 두 페이지가 서로 다른 주장
   - `index.md`에서 유사 태그를 가진 페이지 그룹을 묶어 비교

## Phase 4 — Deprecated/Stale 탐지

빠르게 변하는 도메인의 오래된 페이지를 우선 검토한다.

대상 태그 목록: `AI`, `ML`, `LLM`, `딥러닝`, `프레임워크`, `클라우드`, `보안`, `블록체인`

각 stale 후보 페이지:
- `updated` 날짜가 현재 기준 **180일 이상** 경과한 것
- 본문에 버전 번호나 출시일 같은 구체적 기술 상태 서술이 있으면 **`deprecated 위험 높음`** 등급
- 시점 의존 표현만 있으면 **`deprecated 위험 낮음`** 등급

## Phase 5 — 보고서 출력

항상 아래 형식으로 출력한다. 이슈가 없는 섹션은 "없음"으로 표기한다.

```
## Wiki 검증 보고서 — {오늘 날짜}

### 요약
- 검사 페이지: {N}개
- 발견 이슈: {N}개 (심각 {N} / 경고 {N} / 정보 {N})

### 1. 구조 이슈

#### Orphan 페이지 ({N}건)
- [ ] [[페이지명]] — 어떤 페이지에서도 링크되지 않음

#### 출처 누락 ({N}건)
- [ ] [[페이지명]] — ## 출처 섹션 없음

#### Frontmatter 결함 ({N}건)
- [ ] [[페이지명]] — {누락된 필드} 없음

### 2. Deprecated/Stale 위험

#### 높음 (구체적 기술 상태 서술 + 180일 이상 미갱신)
- [ ] [[페이지명]] — updated: {날짜}, 위험 표현: "{발견된 텍스트}"

#### 낮음 (시점 의존 표현만 있음)
- [ ] [[페이지명]] — 발견된 표현: "{텍스트}"

### 3. 신뢰성·논리성 이슈

#### 근거 없는 단언
- [ ] [[페이지명]] — "{문제 문장}" — 출처 없음

#### 미해결 모순·불확실 표기
- [ ] [[페이지명]] ({라인}): ⚠️/{❓}/{🕰️} 표기 존재, 해결 필요

### 4. 권장 조치
1. 즉시: {가장 심각한 이슈}
2. 이번 주: {중간 우선순위 이슈들}
3. 나중에: {낮은 우선순위 이슈들}
```

## Phase 6 — 수정 진행

보고서를 사용자에게 보여준 뒤 묻는다:
"어떤 항목부터 수정할까요? (번호로 선택 / '전체' / '건너뜀')"

사용자가 승인한 항목에 대해서만 수정을 진행한다.
수정 완료 후 `log.md`에 `lint` 항목을 추가한다:

```
## [{오늘 날짜}] lint | Wiki 품질 검증
- 검사 페이지: {N}개
- 발견 이슈: {N}개
- 수정 완료: {수정한 항목 목록}
```
