# Wiki Restructure

위키의 페이지 간 관계와 폴더 구조를 분석하고 재편성한다.

**절대 원칙: 사용자 승인 없이 파일을 이동하거나 삭제하지 않는다.**
제안 먼저 → 사용자 승인 → 실행 순서를 반드시 지킨다.

## Phase 1 — 연결 그래프 구성

현재 작업 디렉토리의 `wiki/` 아래 모든 `[[wikilink]]`를 수집해 관계 지도를 만든다.

```bash
# 모든 wikilink 수집 및 빈도 집계
grep -rh '\[\[' wiki/ --include="*.md" | \
  grep -oP '(?<=\[\[)[^\]|]+' | sort | uniq -c | sort -rn
```

분석할 지표:
- **허브 페이지**: 5개 이상 다른 페이지에서 링크받는 페이지 → 엔티티나 개념의 중심 역할
- **Orphan 페이지**: 링크 없이 혼자 있는 페이지 → 연결 필요
- **단방향 링크**: A→B는 있는데 B→A가 없는 경우 → 상호 링크 추가 후보
- **과밀 카테고리**: 특정 폴더에 페이지가 20개 이상 집중된 경우 → 하위 카테고리 분리 후보
- **중복 가능성**: 제목이 유사하거나 같은 태그를 3개 이상 공유하는 페이지 쌍 → 병합 후보

## Phase 2 — 재구성 기회 탐지

`index.md`를 읽어 전체 페이지 목록을 파악한 후, 아래 패턴을 탐지한다.

**A. 병합 후보**
같은 주제를 다루는 페이지가 여러 개 있는 경우.

기준 (둘 중 하나 이상):
- 두 페이지가 같은 태그를 3개 이상 공유하면서 내용이 겹침
- 서로를 wikilink로 참조하면서 정의 섹션이 중복됨

예: `concepts/Apache-Kafka.md`와 `entities/Apache Kafka.md`처럼 개념 페이지와 엔티티 페이지가 동일 주제를 다루는 경우

**B. 분리 후보**
한 페이지가 너무 넓은 범위를 커버하는 경우.

기준: `## 상세 설명` 또는 본문 H3 섹션이 4개 이상이고 각 섹션이 서로 독립적인 개념을 다룸

**C. 하위 카테고리 분리 후보**
`concepts/` 또는 `entities/` 안에 같은 도메인의 페이지가 10개 이상 몰린 경우.

예: ML/AI 관련 페이지가 10개 이상 → `concepts/ml/` 하위 디렉토리 분리 검토

**D. 누락 링크**
페이지 본문에서 다른 위키 페이지를 텍스트로 언급하지만 `[[wikilink]]`로 감싸지 않은 경우.

```bash
# 예: "쿠버네티스"라는 텍스트가 있지만 [[쿠버네티스]] wikilink가 없는 파일 탐지
grep -rn "쿠버네티스" wiki/ --include="*.md" | grep -v "\[\[쿠버네티스\]\]"
```

## Phase 3 — 재구성 제안 보고서

사용자에게 제안 목록을 출력한다. 임팩트가 큰 것부터 순서대로.
**이 단계에서는 아무것도 수정하지 않는다.**

```
## Wiki 재구성 제안 — {오늘 날짜}

### A. 병합 제안 ({N}건)
1. [[개념A]]와 [[개념B]] 병합
   - 이유: {겹치는 내용 설명}
   - 결과 파일: [[개념A]] (개념B 내용을 흡수)
   - 영향받는 링크: {N}건

### B. 분리 제안 ({N}건)
1. [[큰개념]] → [[작은개념1]] + [[작은개념2]]로 분리
   - 이유: {H3 섹션들이 독립적인 이유}

### C. 하위 카테고리 생성 제안 ({N}건)
1. `concepts/ml/` 생성 — {대상 페이지 목록}
   - 이동할 페이지: {N}건

### D. 누락 링크 추가 ({N}건)
1. [[페이지A]]에서 [[페이지B]] wikilink 추가
   - 근거: "{언급된 텍스트}"

---
어떤 항목을 진행할까요? (번호로 선택 / "전체" / "건너뜀")
```

## Phase 4 — 실행 (사용자 승인 후만)

각 제안 유형별 실행 절차:

### 병합 실행

1. 두 파일을 모두 읽어 내용을 합친다
2. frontmatter:
   - `tags`: 두 파일의 합집합
   - `sources`: 두 파일의 합집합
   - `updated`: 오늘 날짜
3. 남길 파일에 병합된 내용을 쓴다
4. 삭제할 파일의 이름을 참조하는 모든 wikilink를 남길 파일로 업데이트:
   ```bash
   grep -rn "\[\[삭제할파일\]\]" wiki/ --include="*.md"
   # 각 파일에서 [[삭제할파일]] → [[남길파일]] 치환
   ```
5. 삭제할 파일을 제거하고 `index.md`에서 해당 항목 삭제

### 파일 이동 실행 (하위 카테고리 생성)

1. 새 디렉토리 생성
2. 파일 이동:
   ```bash
   mv wiki/concepts/파일명.md wiki/concepts/ml/파일명.md
   ```
3. 이전 경로를 참조하는 wikilink 일괄 업데이트:
   ```bash
   # 이전 파일명으로 링크된 것 찾기
   grep -rn "\[\[파일명\]\]" wiki/ --include="*.md"
   # wikilink는 경로가 아닌 파일명으로 참조하므로
   # 파일명이 동일하면 이동 후에도 대부분 정상 동작
   # Obsidian은 자동으로 경로를 해석함
   ```
4. `index.md`에서 경로 업데이트
5. 이동된 파일의 frontmatter `updated` 날짜 갱신

### 누락 링크 추가

해당 파일을 열어 텍스트를 `[[wikilink]]` 형식으로 감싼다.
예: `쿠버네티스를 사용하면` → `[[쿠버네티스]]를 사용하면`

## Phase 5 — 완료 보고

실행한 변경 사항을 요약하고 `log.md`에 `meta` 항목을 추가한다:

```
## [{오늘 날짜}] meta | Wiki 구조 재편성
- 병합: {건수}건 (예: [[A]] + [[B]] → [[A]])
- 이동: {건수}건 (예: concepts/{N}개 → concepts/ml/)
- 링크 추가: {건수}건
- index.md 갱신
```
