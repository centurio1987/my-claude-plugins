# 사후 작업 검토 (post-work-review)

PLAN.md / BACKLOG.md 로 진행한 작업을 마친 뒤, **에이전트와 함께 만든 최종 산출물**을 분석해
거기서 식별 가능한 세 종류의 후보를 찾아 제안하고, 사용자가 고른 것만 실제 파일로 만든다.

- (A) **반복·식별된 작업** → 재사용할 **스킬** 후보
- (B) **하위 디렉토리의 `CLAUDE.md`** → **프로젝트 스코프 룰** 전환 후보
- (C) **병렬화 가능한 작업** → **에이전트** 후보

> 분석 대상은 "플러그인 자신"이 아니라 **사용자가 방금 끝낸 작업 결과물**(레포·코드·문서·디렉토리 구조)이다.

생성물 저장 위치는 **타겟 프로젝트의 `.claude/`** 이다(스킬→`.claude/skills/`, 에이전트→`.claude/agents/`,
룰→`.claude/CLAUDE.md`). 파괴적 이동·자동 삭제는 하지 않는다.

---

## Phase 1 — 작업 맥락 수집

이번에 무엇을 만들었는지(산출물 범위) 파악한다. **토큰 폭증을 막기 위해 전체 diff는 통으로 읽지 않는다.**

1. `PLAN.md` / `BACKLOG.md` 의 완료 항목(`- [x]`)을 읽어 "무엇을 하기로 했고 끝냈는지" 정리한다.
2. git 레포라면 범위 파악은 **요약 위주**로:
   - `git log --oneline -20`
   - `git diff --stat`(작업 기준점 대비; 모르면 최근 커밋들 기준)
   - 변경/추가된 파일 **목록**
3. 위 목록에서 의심 가는 파일만 **선택적으로** Read 한다. 대규모 라이브러리/생성 파일은 건너뛴다.

## Phase 2 — 산출물 분석 (work-pattern-analyzer 에이전트 1회 호출)

`work-pattern-analyzer` 에이전트를 **한 번** 호출하여, Phase 1 에서 정리한 산출물 범위를 넘기고
세 카테고리 후보를 **한 번에** 받아온다(내부 병렬 fan-out 하지 않는다 — 단순·견고).

에이전트에 전달할 것:
- 작업 요약(완료 항목), 변경 파일 목록, 분석 루트 디렉토리.
- 요청: "아래 세 종류 후보를 고정 스키마로 반환하라" — (A) 스킬, (B) 하위 CLAUDE.md 룰, (C) 에이전트.

기대 반환(후보당): `type`(skill|rule|agent), `name`, `evidence`(경로 중심), `rationale`, `confidence`(low|med|high).

**Fallback** — 에이전트 호출이 실패하거나 반환 형식이 깨졌으면, 스킬이 직접 최소 분석을 수행한다:
- (A) 반복 패턴: 비슷한 스크립트/명령/문서 작업이 여러 번 나타나는지 Grep/Glob 로 점검.
- (B) `Glob **/CLAUDE.md` 로 하위 디렉토리 CLAUDE.md 탐색(루트 CLAUDE.md 제외).
- (C) 서로 독립적이라 동시에 돌릴 수 있는 작업 군집이 있는지 점검.
- 이 경우 리포트 상단에 **"⚠ 에이전트 분석 실패 — 축약 결과"** 를 명시한다.

> 프라이버시: 근거는 **파일 경로 위주**로 인용한다. 비밀키·토큰·개인정보(PII)로 보이는 내용은 인용하지 않는다.

## Phase 3 — 제안 리포트 출력

세 카테고리를 표로 정리해 사용자에게 보여준다. 후보가 없으면 "해당 없음"으로 명시한다.

```
## 사후 작업 검토 결과

### A. 스킬 후보 (반복·재사용 가능 작업)
| 제안 이름 | 근거(경로) | 왜 | 신뢰도 |
|---|---|---|---|
| ... | ... | ... | high/med/low |

### B. 프로젝트 룰 후보 (하위 CLAUDE.md → project scope rule)
| 대상 CLAUDE.md | 핵심 지시 요지 | 승격 제안 위치 | 신뢰도 |
| ... | ... | .claude/CLAUDE.md | ... |

### C. 에이전트 후보 (병렬화 가능 작업)
| 제안 이름 | 병렬화 근거 | 왜 | 신뢰도 |
| ... | ... | ... | ... |
```

## Phase 4 — 사용자 선택

`AskUserQuestion`(multiSelect)으로 **실제로 생성할 후보**만 고르게 한다. 아무것도 안 골라도 된다.
후보가 많으면 카테고리별로 나눠 묻는다.

## Phase 5 — 생성 실행 (초안 생성 + 확인)

사용자가 고른 것만, **타겟 프로젝트 `.claude/`** 아래에 생성한다.

- **스킬 후보** → `<project>/.claude/skills/<name>/SKILL.md`
  - `skill-creator` 스킬이 사용 가능하면 그것으로 스캐폴드한다.
  - 없으면 **표준 SKILL.md 템플릿(frontmatter `name`/`description` + 단계)을 직접 Write** 한다(하드 의존성 없음).
- **에이전트 후보** → `<project>/.claude/agents/<name>.md`
  - frontmatter(`name`, `description` + example), 도구는 작업 성격에 맞게. 초안으로 작성.
- **룰 후보** → 하위 `CLAUDE.md` 내용을 `<project>/.claude/CLAUDE.md` 로 **복사/승격하는 초안**을 제시.
  - **원본 CLAUDE.md 는 자동으로 삭제·이동하지 않는다.** 승격(이동)이 필요하면 사용자에게 한 번 더 확인.

**안전장치**:
- 생성 대상 경로에 **동일 이름 파일이 이미 있으면 덮어쓰지 말 것.** 경고하고 다른 이름을 제안하거나 사용자 확인을 받는다.
- v0.1.0 범위는 여기까지(초안 + 확인). 일괄 자동 이동/삭제는 하지 않는다.

마지막으로 생성된 파일 목록과 다음 행동(예: 새 스킬을 쓰려면 세션 재시작/마켓 등록)을 한 줄로 안내한다.
