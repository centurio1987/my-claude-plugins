# research 스킬 (리서치 오케스트레이터)

주제를 받아 **웹에서 근거를 모아 충실히 정리한 한국어 문서**를 만든다. 핵심 최적화는 **수집을
서브에이전트(`researcher`)로 격리**하는 것이다 — `WebFetch` 본문 덤프가 메인 대화를 오염시키지
않도록, 워커는 정제된 구조화 findings만 반환하고 이 스킬은 그걸 병합해 최종 문서를 쓴다.

> **이 스킬(메인 스레드)은 긴 수집 규칙을 읽지 않는다.** 상세 규칙(RESEARCH_GUIDE)은 워커에게만
> 로드된다. 메인은 짧은 `OUTPUT_FORMAT.md`만 참조해 병합·작성한다(컨텍스트 절감).

## 입력 / 출력 계약

- **입력**: 조사할 주제 또는 글감(자유 텍스트).
- **출력**: `<outputDir>/<YYYY-MM-DD>-<slug>.md` — 출처 추적 가능한 한국어 리서치 문서 1개.
- **불변 원칙**: 출처 없는 정보·추측·창작 금지. 광고성 콘텐츠 제외. 정보 충돌 시 양쪽 다 기록.
  모르는 것은 "open question"으로 남긴다.

## 동작 순서

### 1. 산출물 저장 위치 확보 (최초 실행 시 질문)

`~/.config/web-research/config.json`의 `outputDir`를 읽는다.

- **파일/키가 있으면** 그 경로를 그대로 쓴다(이후 재질문하지 않음).
- **없으면**: 워커를 디스패치하기 **전에 즉시 멈추고**, 사용자에게 "리서치 결과를 어디에 저장할까요?"를
  묻는다(`AskUserQuestion` 또는 일반 질문). 답을 받으면 `~/.config/web-research/config.json`에
  `{"outputDir": "<경로>"}`로 저장하고, **사용자가 다시 요청하면 그 시점부터** 수집을 진행한다.
- 사용자가 "저장 위치 바꿔줘"라고 하면 config를 갱신한다.
- **쓰기 폴백**: config 디렉토리 쓰기가 실패하면 ① 기본값 `~/research-notes/`를 쓰거나 ② 사용자에게
  매번 경로를 인자로 받는다. `outputDir`가 존재하지 않으면 `mkdir -p`로 생성한다.

### 2. 주제 분해 & 팬아웃 범위 판단

주제를 소주제로 쪼개고, 워커 수(1~4)를 **적응적으로** 정한다.

- 단일 사실·정의·단일 제품/개념 → **1개**.
- 비교·시장 조사·기술 동향·다면 주제 → 소주제당 1워커로 **2~4개** 병렬.
- 역사 + 현황이 섞인 주제 → 과거(기원·맥락)와 현재(최신 동향)를 분리.
- **시간 민감 주제**(가격·API·법률·인물·버전·정치) → 최신 확인을 최우선. **역사·학술 배경** → 1차 원전 우선.

### 3. researcher 서브에이전트 병렬 디스패치

`Agent` 도구로 `subagent_type: researcher`를 (가능하면 한 메시지에 여러 개) 디스패치한다. 각 프롬프트에 포함:

- 담당 **소주제**와 전체 주제 맥락.
- **산출물 언어: 한국어**.
- "**먼저 `research-method` 스킬을 호출**해 수집 규칙·품질기준·출력계약을 로드하라(실패 시
  플러그인의 `RESEARCH_GUIDE.md`를 직접 Read)".
- **반환 형식**: 구조화 findings(소주제 정리 + open questions + 주장↔소스 매핑 + 소스 목록). 파일은 쓰지 말 것.
- **반환 길이 상한**: 원문 전체 덤프 금지, 정제된 내용만. 워커는 자기 소스에 임시 번호 `(S1)(S2)…`를 붙인다.
- **레이트리밋/실패 폴백**: 병렬 `WebFetch`가 rate-limit/실패하면 1회 재시도, 그래도 실패면 순차로 돌리거나
  미수집분을 open question으로 보고. 워커 일부가 죽어도 나머지로 진행한다.

### 4. 병합 & 전역 소스 재매핑

워커별 findings를 모아 하나로 합친다.

- **전역 소스 번호 재매핑**: 워커마다 중복된 `(S1)(S2)`를 전역 고유번호(S1…SN)로 안전하게 치환한다.
  워커별 임시번호 → 전역번호 매핑표를 만든 뒤, 각 워커 본문의 인라인 표기를 그 매핑대로 일괄 치환한다
  (인용 링크가 깨지지 않게).
- **중복 소스 제거**: URL 정규화(UTM·추적 파라미터 제거, http/https·`www` 정규화) 후 동일 판단.
  같은 기사 재배포, PDF/HTML 동일 자료, 기관 페이지와 보도자료 중복을 하나로 묶는다.
- **사실 통합**: 같은 사실을 말하는 소스는 묶고, **충돌하는 정보는 양쪽 다 기록**한다(어느 쪽이 더 신뢰되는지
  근거와 함께 — 단 임의 단정 금지).
- 전체 문서 기준 **신뢰 소스 최소 5개**를 확보했는지 확인(부족하면 추가 워커/검색 또는 open question 명시).

### 5. 최종 문서 작성

`research-method`의 `assets/OUTPUT_FORMAT.md` 형식대로 **한국어** 문서를 `<outputDir>/<YYYY-MM-DD>-<slug>.md`로 `Write`한다.

- **날짜**: KST(Asia/Seoul) 기준 `YYYY-MM-DD`.
- **slug**: 한글 주제는 영문 키워드/음차로 slug화, 특수문자 제거, 적당한 길이로 자른다.
- **파일명 충돌**: 이미 있으면 `-2`, `-3` … suffix를 붙인다.

### 6. 종료 보고

- 만든 파일 경로, 확인한 신뢰 소스 수, 주요 open questions, 충돌로 양쪽 기록한 항목.

## 참조 파일

- `../research-method/assets/OUTPUT_FORMAT.md` — 최종 문서 형식(메인이 참조).
- `../research-method/assets/RESEARCH_GUIDE.md` — 수집 규칙(워커가 로드, 메인은 읽지 않음).
- `agents/researcher.md` — 워커 서브에이전트 정의.

## 주의

- 출처 없는 정보·추측·창작 금지. 광고성 제외. 충돌은 양쪽 기록. 모르면 open question.
- 수집의 무거운 작업은 **반드시 워커에 위임**한다 — 메인 스레드에서 직접 `WebFetch`를 남발하지 않는다.
- 외부 작업(웹 접근)은 실패할 수 있다 — 실패는 숨기지 말고 open question/보고로 남긴다.
