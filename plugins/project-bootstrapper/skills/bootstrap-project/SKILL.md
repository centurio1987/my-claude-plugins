# Bootstrap Project (오케스트레이터)

대상 프로젝트에서 작업하게 될 에이전트가 **스스로 참고할 산출물**을 갖추도록 부트스트랩한다.
소통 창구는 단일 파일 `<관리폴더>/PLAN.md`이며, 이 스킬은 그 PLAN.md를 함께 완성해 나가는 과정을 주도한다.

## 핵심 원칙

- **PLAN.md 단일 창구**: 부트스트랩(운영 방식 정의) 단계에서 유저가 직접 편집·갱신하는 파일은 PLAN.md **하나뿐**이다. WORKFLOW/BACKLOG/WBS/CLAUDE.md 등 나머지 산출물은 모두 에이전트가 PLAN.md에 근거해 생성·관리한다.
- **ORDER.md는 별개의 상시 지시 창구**: PLAN.md가 *운영 방식을 정의하는 1회성 인터랙티브 창구*라면, `<project-root>/ORDER.md`는 *프로젝트 운영 중 문서로 지시를 받는 상시 창구*다. ORDER.md와 그 처리 스킬(`process-order`)은 `materialize-bootstrap` 단계에서 자동 생성된다(유저가 설계할 항목 아님).
- **claude-native 우선**: 에이전트가 참고할 산출물은 임의 파일이 아니라 가능한 한 claude 도구(skill·command·agent·`CLAUDE.md`)로 만들어 `<project-root>/.claude/*`에 둔다. 거버넌스 문서(WORKFLOW/BACKLOG/WBS/PLAN)는 사람이 찾기 쉬운 최상위 관리폴더(기본 `manage/`)에 둔다.
- **점진적 공동 완성**: 스켈레톤 제공 → 유저 초안 → 에이전트 리뷰·발전. 한 번에 완성하지 않는다.

## Phase 0 — 사전 점검

1. 대상 프로젝트 루트 디렉토리를 확인한다(기본값: 현재 작업 디렉토리). 다른 경로면 유저에게 확인한다.
2. 기존 산출물 존재 여부를 점검한다:
   - `manage/PLAN.md`(또는 다른 관리폴더의 PLAN.md)가 이미 있으면 → **이어서 보완 모드**. 스켈레톤을 덮어쓰지 말고 기존 PLAN.md를 읽어 Phase 2로 진입한다.
   - `.claude/` 또는 `manage/`에 산출물이 이미 있으면 유저에게 알리고, 새로 시작할지/이어갈지 확인한다.
3. 관리폴더명을 정한다(기본 `manage`). 유저가 선호 폴더명을 제시하면 그대로 쓴다.

## Phase 1 — 스켈레톤 제공

1. `templates/PLAN.skeleton.md`를 읽어 `<관리폴더>/PLAN.md`로 기록한다(관리폴더가 없으면 생성). 날짜·관리폴더명 등 자명한 값은 채워 둔다.
2. 유저에게 다음을 안내하고 **이번 턴을 종료한다**:
   - "`<관리폴더>/PLAN.md`에 항목 위주 스켈레톤을 만들었습니다. 각 섹션의 초안을 직접 작성/수정한 뒤 알려주세요. 비워둬도 되는 부분은 그대로 둬도 됩니다 — 제가 이어서 함께 구체화하겠습니다."
   - Claude Code는 턴 기반이므로 watcher/폴링은 필요 없다. 유저가 편집 후 메시지를 보내면 자연스럽게 세션이 재개된다.

## Phase 2 — 초안 확인

유저가 재개하면 `<관리폴더>/PLAN.md`를 **다시 읽어** 현재 상태를 파악한다.
- 비어 있거나 모호하거나 서로 모순되는 섹션을 식별한다.
- 특히 워크플로우 phase 정의, 백로그/WBS 선택, 산출물 정책(어떤 필요를 어떤 claude 도구로 만들지)이 실행 가능한 수준인지 점검한다.

## Phase 3 — 리뷰 & 발전

`AskUserQuestion`으로 섹션별 구체화 질문을 던지며 `PLAN.md`를 함께 편집한다(Edit로 직접 PLAN.md를 갱신).
한 번에 1~3개 질문씩, 다음이 모두 실행 가능한 수준이 될 때까지 반복한다:

- **산출물 정책**: 각 "필요"가 어떤 claude 도구로 매핑되는가(반복 절차→skill, 지식→CLAUDE.md/참고문서, 자동화→hook/command).
- **워크플로우**: 포함 여부. 포함 시 각 phase의 목적·입력·절차·산출물·DoD가 명확한가. 절차가 구체적인 phase는 `skill화: yes`로 표시.
- **백로그/WBS**: 포함 여부와 형식(BACKLOG 또는 WBS), 초기 항목.
- **규칙·컨벤션·협업 톤**.

각 라운드 후 갱신된 PLAN.md의 변경점을 한 줄로 보고한다.

## Phase 4 — 확정 & 인계

1. PLAN.md의 모든 핵심 섹션이 채워졌는지 유저와 함께 최종 확인한다.
2. PLAN.md 상단(또는 메타 섹션)에 `상태: 확정(YYYY-MM-DD)`을 기록한다.
3. `materialize-bootstrap` 스킬을 **Skill 도구로 호출**해 산출물 생성을 인계한다.
   - 연쇄 호출이 동작하지 않으면 폴백으로 유저에게 안내한다:
     "PLAN.md가 확정되었습니다. `/project-bootstrapper:materialize-bootstrap` 를 실행하면 manage/ 문서와 .claude/ 도구를 생성합니다."
